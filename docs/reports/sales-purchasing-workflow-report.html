1. Business Process Overview

The system runs two parallel commercial workflows that both ultimately move physical stock and money: the Sales side (selling stock, collecting receivables from customers) and the Purchasing side (buying stock, paying payables to suppliers).

Sales sidePurchasing side
CounterpartyCustomerSupplier
GoalSell stock, collect receivablesBuy stock, pay payables
Pre-invoice documentSales Order → Delivery NotePurchase Order → Goods Receipt
Invoice documentSales InvoicePurchase Invoice
Return documentSales Credit NotePurchase Credit Note
Stock directionDecreases (SALE_OUT), increases on return (RETURN_IN)Increases (PURCHASE_IN), decreases on return (RETURN_OUT)

Both sides share one Invoice table (type: SALE | PURCHASE) and one CreditNote table (type: SALE | PURCHASE) — they are the same schema, branched by type, not separate models.

Purchase OrderDRAFT → SENT → PARTIAL → RECEIVED, or CANCELLED

2. Sales Workflow

2.1 Sales Order Lifecycle

SalesOrderStatus: DRAFT → CONFIRMED → PARTIAL → FULFILLED, or CANCELLED at DRAFT/CONFIRMED.

  • Create (createSalesOrderAction): status DRAFT; lines store displayQuantity/baseQuantity/unitPrice/discount; deliveredBaseQuantity = 0.
  • Confirm (confirmSalesOrderAction): runs the credit-limit check (see 2.5). No stock movement here — confirming an SO never touches inventory.
  • Deliver (createDeliveryNoteAction): the only place SO stock actually leaves the warehouse. Recomputes status to PARTIAL (some lines short) or FULFILLED (all lines fully delivered).
  • Cancel: only DRAFT/CONFIRMED, and only if no delivery notes exist yet.

2.2 Delivery Note Lifecycle

No own status field — it is an append-only event tied to a Sales Order. Each DeliveryNoteLine references a SalesOrderLine and carries the dispatched quantity. Creating one:

  1. Validates dispatched qty ≤ remaining (baseQuantity - deliveredBaseQuantity).
  2. Calls recordMovement(SALE_OUT) per line — this is where stock actually decreases.
  3. Increments SalesOrderLine.deliveredBaseQuantity.
  4. Recomputes the parent SO status.

2.3 Sales Invoice Lifecycle

InvoiceStatus: DRAFT → CONFIRMED → CANCELLED.

  • Confirm (confirmSalesInvoiceAction): checks stock sufficiency and calls recordMovement(SALE_OUT) per line, then sets dueDate from the customer's paymentTerms.
  • Cancel: any DRAFT/CONFIRMED invoice; stock is not automatically reversed on cancel.
Design risk — double stock movement: this is a second, independent path to SALE_OUT, separate from the SO → Delivery Note path. A direct walk-in sale (no SO) decrements stock at invoice confirmation. An SO-driven sale decrements stock at the Delivery Note step. There is no Invoice ↔ DeliveryNote link in the schema and no guard preventing a manually created invoice from re-decrementing stock that an SO's delivery note already moved.

2.4 Sales Credit Note / Return Lifecycle

CreditNoteStatus: DRAFT → CONFIRMED / CANCELLED (CONFIRMED is immutable).

  • Created only against a CONFIRMED sales Invoice (originalInvoiceId), with lines pre-filled from the invoice and quantity-capped at "invoiced minus already-credited" — matched by productId + unitId, since CreditNoteLine has no direct invoice-line foreign key.
  • Confirm: recordMovement(RETURN_IN) per line — restocks the warehouse — then becomes immutable.
  • Cancel: DRAFT only.

2.5 Customer Balance & Credit-Limit Logic

Both the customer detail page and the Sales-Order credit check use the same formula:

outstandingBalance = totalInvoiced(confirmed SALE invoices)
                    - totalPaid(payments on those invoices)
                    - totalCredited(confirmed SALE credit notes,
                                    value = Σ displayQuantity × unitPrice)

Credit check on SO confirm: outstandingBalance + thisSOValue > customer.creditLimit → blocked with an error showing all three numbers and the limit. creditLimit == null means no check (unlimited).

3. Purchasing Workflow

3.1 Purchase Order Lifecycle

PurchaseOrderStatus: DRAFT → SENT → PARTIAL/RECEIVED, or CANCELLED at DRAFT/SENT.

  • Create: status DRAFT, lines carry unitCost, receivedBaseQuantity = 0.
  • Mark Sent: DRAFT → SENT, sets sentAt.
  • Cancel: DRAFT/SENT only.

3.2 Goods Receipt Process

This is where purchasing stock actually arrives:

  1. Only against SENT/PARTIAL POs.
  2. Per line, validates received qty ≤ outstanding (baseQuantity - receivedBaseQuantity).
  3. recordMovement(PURCHASE_IN) per line.
  4. Increments PurchaseOrderLine.receivedBaseQuantity.
  5. Recomputes PO status: all lines fully received → RECEIVED (+ receivedAt); some → PARTIAL.

3.3 Purchase Invoice Lifecycle GAP

DRAFT → CONFIRMED → CANCELLED, optionally linked to a PO via Invoice.purchaseOrderId (only allowed once that PO is PARTIAL/RECEIVED).

Confirming a Purchase Invoice also calls recordMovement(PURCHASE_IN) per line — independently of whatever the Goods Receipt already moved. There is no quantity-matching guard between "already received via Goods Receipt" and "this invoice's lines." If a PO-linked invoice is confirmed for the same goods a Goods Receipt already recorded, stock is double-counted.

3.4 Purchase Credit Note / Return Lifecycle

Same shape as the sales side: DRAFT → CONFIRMED/CANCELLED, created against a CONFIRMED purchase Invoice; confirm → recordMovement(RETURN_OUT) (stock leaves, going back to the supplier), then immutable.

3.5 Supplier Balance GAP

balance = totalInvoiced(confirmed PURCHASE invoices) - totalPaid
This does not subtract confirmed PURCHASE credit notes. It is the exact gap that existed on the customer side before being fixed this session — a supplier return currently lowers stock but does not lower what the system shows as owed to that supplier. Same fix pattern (subtract Σ creditNote.lines.displayQuantity × unitPrice for confirmed PURCHASE credit notes) would close it.

4. Inventory Impact

All stock-affecting actions funnel through one function, recordMovement() (core/inventory/record-movement.ts), which atomically updates InventoryBalance.currentQuantity and appends an immutable InventoryMovement row, then fires low-stock / reorder-point notifications.

Document / ActionMovement typeStock effect
Delivery Note createdSALE_OUTDecreases
Sales Invoice confirmed (no prior delivery)SALE_OUTDecreases
Sales Credit Note confirmedRETURN_INIncreases (return from customer)
Goods Receipt createdPURCHASE_INIncreases
Purchase Invoice confirmedPURCHASE_INIncreases ⚠️
Purchase Credit Note confirmedRETURN_OUTDecreases (return to supplier)
Stock TransferTRANSFER_OUT / TRANSFER_INMoves between warehouses
Manual adjustmentADJUSTMENTSign follows input

This centralization is a strength: every stock-affecting action is auditable from one trail and shares one negative-balance guard (allowNegative defaults to false).

5. Financial Impact

DocumentAffects
Sales Invoice (confirmed)Receivables (+)
Payment (against sales invoice)Receivables (−)
Sales Credit Note (confirmed)Receivables (−)
Purchase Invoice (confirmed)Payables (+)
Payment (against purchase invoice)Payables (−)
Purchase Credit Note (confirmed)Payables (−) not wired in yet

The outstanding-balance formula is identical in shape on both sides (invoiced − paid − credited), but only the sales side currently includes the credit-note term in code.

The AR Aging report computes balance as originalAmount − paidAmount per invoice, without crediting any sales-credit-note value against it — so the report and the customer-detail page can disagree once returns exist. Worth reconciling onto one shared helper rather than three separate inline formulas (customer page, SO credit check, AR aging export).

8. Current Implementation Status

Fully implemented and consistent DONE

  • PO → Goods Receipt → status recompute
  • SO → Delivery Note → status recompute
  • Both Credit Note flows (create / confirm / cancel) with correct movement direction
  • Centralized recordMovement() for all stock changes, plus low-stock notifications
  • Customer balance correctly nets credit notes
  • AR Aging, Stock Valuation, and Low-Stock reports all implemented

Gaps / technical debt for the roadmap OPEN

  1. Purchase-invoice / goods-receipt double stock-increment risk — no guard against confirming a PO-linked invoice for quantities already received via Goods Receipt.
  2. Supplier balance does not subtract purchase credit notes — mirrors a bug already fixed on the customer side; same fix pattern needed.
  3. AR Aging report's balance formula does not account for credit notes — will drift from the customer-detail balance once returns are used.
  4. No Invoice ↔ DeliveryNote link in schema — makes it impossible to mechanically prevent the sales-side equivalent double-SALE_OUT risk.
  5. Manager / Cashier / Accountant role templates have zero seeded permissions — a functional gap if anyone assigns those roles expecting default access.
  6. CreditNoteLine has no direct foreign key to the source invoice line — return-quantity tracking relies on matching productId + unitId, which could misattribute returns if an invoice ever has two lines for the same product+unit (currently prevented only by convention, not a database constraint).
